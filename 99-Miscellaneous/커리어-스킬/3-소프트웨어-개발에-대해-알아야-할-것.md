# 3부 소프트웨어 개발에 대해 알아야 할 것

아는 것에도 세 가지 단계가 있다.

1. 안다는 걸 안다.
2. 모르는 걸 안다.
3. 모르는 걸 모른다.

개발자가 되기 위해 알아야 하는 것은 충격적일 만큼 많다. 모르는 줄도 모르는 영역을 최대한 없앨 필요가 있다.

## 21. 프로그래밍 언어 개요

- C - 세계에서 가장 많이 쓰인 언어이다. C는 저수준 언어이자 까다로운 언어다.
- C++ - C의 상위 집합이다. C를 확장하여 객체지향성, 클래스, 가상 함수 같은 기능이 추가되었다.
- C# - 마이크로소프트가 닷넷 런타임 환경을 대표하는 언어로 만들었다. 자바와 매우 유사하다.
- 자바 - 한 번 작성한 코드를 여러 플랫폼에서 실행할 수 있게 만든 언어다. C#보다 오래된 언어이다.
- 파이썬 - 가독성을 중요하게 고려해서 만든 언어이다. 초보가 배우기에도 좋다.
- 루비 - 일본에서 만든 객체지향적인 스크립트 언어이다. 루비 온 레일스 덕분에 크게 성장했다. 초심자가 배우기에 아주 좋은 언어이다.
- 자바스크립트 - 10일 만에 개발된 언어다. 문제가 많은 언어였지만 최신 버전에서는 기존 단점을 많이 수정했다.
- 펄 - 유닉스 스크립트용으로 많이 쓰인다. 가독성이 아쉽다.
- PHP - 유명한 웹사이트 중에는 처음에 PHP로 시작한 곳이 많다. 지금까지도 쓰는 곳이 있다. 길을 잃기 쉬운 후미진 골목이 많이 도사리고 있는 언어이다.
- 오브젝티브-C - C에 객체지향 기능을 더하기 위해 만들어진 언어이다. 애플은 맥 OS X 운영 체제에 이 언어를 사용했다.
- 스위프트 - 애플이 선택한 새로운 iOS용 대표 언어다. iOS 개발을 할 생각이라면 오브젝티브-C를 거르고 바로 스위프트로 시작해도 좋다.
- Go - 구글이 비교적 최근에 만든 언어다. C와 매우 비슷하지만 몇 가지 추가 기능이 있다. 가비지 컬렉션, 동시성 프로그래밍 기능이 있다.
- 얼랭 - 동시 및 분산 처리를 위해 설계된 언어이다. 코드 핫 스와핑을 지원하여 애플리케이션 종료 없이 코드를 변경할 수 있다. 가장 안정적인 프로그래밍 언어이자 프로그래밍 환경으로 여겨진다.
- 하스켈 - 본래 학구적인 성격이 강한 순수 함수형 프로그래밍 언어이다. 배우기도, 사용하기도 어렵지만 매우 강력하다. 부작용 없이 예측 가능한 코드를 만들 수 있다.

## 22. 웹 개발이란 무엇인가?

**웹 개발을 할 때는 런타임 환경에 대한 통제력이 떨어진다.** 웹 브라우저 세계에는 익스플로러, 에지, 크롬, 사파리, 파이어폭스라는 다섯 강자가 존재한다. 게다가 사람들이 쓰는 브라우저는 모든 버전이 다르다.

소프트웨어 개발자라면 웹 개발의 기본이라도 알아야 한다. **웹 개발은 세상을 점령한 개발 플랫폼계의 킹콩이다.**

웹 개발은 웹 브라우저에서 작동하는 애플리케이션을 만드는 것이다. 어떤 방식을 사용하든 기본 기술은 HTML, CSS, 자바스크립트로 똑같다.

기본적으로 웹 브라우저가 요청을 보내고 웹 서버가 HTML, CSS, 자바스크립트를 반환하는 방식으로 응답한다.

바탕이 되는 HTTP 프로토콜에는 상태가 없으므로 이를 극복할 방법을 찾아야 한다.

**웹의 역사는 정적인 웹 페이지부터 시작해서 페이지에 상호작용을 추가하는 과정으로 볼 수 있다.**

HTML은 웹 페이지의 내용과 구조를 만드는 데 쓰인다.

웹 페이지의 스타일을 정의하기 위해 웹 페이지에 CSS를 연결할 수 있다. CSS 전처리기를 사용하면 변수를 할당하고 반복문을 실행하고 CSS 대부분을 재사용할 수 있다.

자바스크립트는 웹 브라우저에서 바로 실행되는 완전한 함수형 동적 언어다. 웹 페이지와 콘텐츠를 프로그램으로 관리할 수 있다.

서버 측 렌더링은 웹 페이지가 서버에서 만들어진다. 클라이언트 측 렌더링은 자바스크립트를 사용해 웹 페이지를 서버가 아닌 브라우저에서 만든다.

웹 개발자가 되려면 API를 이해해야 한다. API는 어떤 프로그램이 다른 프로그램에 동작이나 데이터를 요청하기 위해 전송하는 명령어 명세다. SPA는 대개 API를 통해 데이터를 주고받는다.

## 23. 모바일 개발

모바일 개발은 아주 흥미로운 분야로 늘 손꼽힌다. 개발자 혼자서도 단기간 내에 앱을 개발할 수 있는 특별한 기회를 제공하기 때문이다.

모바일 기기를 위해 개발하는 것이라면 전부 모바일 개발이라고 보아야 한다.

iOS는 BSD 계열인 다윈과 M을 기반으로 하고 있어 근본적으로 유닉스와 유사하다. iOS는 맥 OS와 주요 프레임워크 일부를 공유하다. iOS 애플리케이션은 오브젝티브-C나 스위프트를 써서 만드는데 요즘 더 인기를 끄는 쪽은 스위프트다.

안드로이드 OS 애플리케이션 네이티브 개발 플랫폼은 자바다.

모바일 앱을 네이티브로 개발하려면 iOS용 코드와 안드로이드용 코드를 완전히 다르게 써야 한다. 도구, 언어, 개발 패러다임도 다르다. 유지 보수가 매우 힘들다. 하지만 네이티브 개발은 가장 큰 장점은 속도다.

크로스 플랫폼 프레임워크 솔루션을 선택할 때 고려해야 할 사항은 다음과 같다.

- 프로그래밍 언어
- 네이티브 vs. 하이브리드
- 지원하고자 하는 플랫폼의 종류
- 코드 재사용

모바일 웹 앱이 미래라는 의견도 있지만, 시간이 좀 더 필요해 보인다.

## 24. 백엔드 개발

백엔드 개발은 사용자 인터페이스를 만들어내는 코드 작성과 관련 없는 모든 개발이다. 간단히 말해 보이지 않는 영역에 있는 코드를 작성하는 것이다.

백엔드 개발자는 프런트엔드 개발자에 비해 알고리즘을 적용하고 문제를 해결하는 데 더 많은 시간을 쓴다.

백엔드 개발자는 애플리케이션 아키텍처를 제대로 알아야 한다.

풀스택 개발에서는 시스템의 모든 컴포넌트와 계층을 다룬다. 서버 하드웨어와 아키텍처뿐 아니라 데브옵스 부분까지 알아야 한다.

**기술 스택 전반에 대해 일반적인 지식을 갖추되 일부 영역에 대해 전문성을 갖는 게 현명하다.**

## 25. 비디오 게임 개발

비디오 게임 개발 분야는 경쟁이 매우 치열하다. **비디오 게임 개발자가 되기 싫은 사람이 있을까?** 프로그래머들 중 70퍼센트는 살면서 한 번쯤 비디오 게임 개발자에 대한 환상을 품어본 적이 있을 것이다.

비디오 게임 개발자의 근무시간은 지나치게 긴 경우가 태반이다.

비디오 게임 개발자가 되려면 학위를 받거나 직업 교육 프로그램이라도 이수하기를 추천한다.

대형 게임 스튜디오가 게임 개발에 쓰는 주요 언어는 여전히 **C++**이다.

비디오 게임 개발자라면 **비디오 게임 엔진**도 경험해보아야 한다. 유니티, 언리얼 엔진이 있다.

게임 개발자에게는 **수학**도 매우 중요한 기술이다.

게임 개발을 하고자 한다면 게임을 최대한 많이 개발해볼 것을 강력히 추천한다.

## 26. DBA와 데브옵스

DBA 업무를 개발자에게 맡기는 조직이 많다. DB 관련해서 기본적으로 배워두면 좋은 항목은 다음과 같다.

- 데이터베이스 설치 및 설정하기
- 백업 만들고 복구하기
- 테이블과 스키마 만들기
- 저장 프로시저 만들기
- 테이블 인덱스 만들기와 인덱스 작동법
- CRUD와 같은 기초적인 SQL 코드 작성하기
- 테이블 조인

데브옵스는 개발과 운영의 복합체 혹은 매시업이라고 볼 수 있다. 코드를 개발해서 생산하기까지의 각 단계를 관리하는 종합적인 공정을 가리킨다. 소프트웨어 개발자는 엄밀히 말해 데브옵스에 속한다.

## 27. 소프트웨어 개발 방법론

**소프트웨어 개발 방법론과 이를 구현할 방법이라는 주제보다 더 많은 논쟁을 불러일으키는 주제는 없을 것이다.**

소프트웨어 개발 방법론은 소프트웨어 제작 절차를 정의한다. 소프트웨어를 만드는 방법, 팀을 이끄는 방법을 구체적으로 알려준다.

### 전통적인 폭포수 개발

폭포수 개발은 이름 그대로다. 물이 바닥을 보일 때까지 한 단계가 끝나면 다음 단계로 넘어간다. 폭포수 개발 방법론에는 소프트웨어 개발 생명주기(Software Development Life Cycle, SDLC)가 포함되어 있다. 폭포수 개발은 SDLC를 점진적으로 따라갈 뿐이라고 보아도 무방하다.

### SDLC

SDLC의 각 단계는 다음과 같다.

1. 요구사항 분석
2. 소프트웨어 디자인
3. 구현
4. 테스트
5. 배포
6. 유지 보수

### 애자일

애자일은 소프트웨어 개발 업계의 판도를 뒤바꿔놓았다. 애자일이 무엇인지는 선언문보다 [원칙](https://agilemanifesto.org/iso/ko/principles.html)이 더 잘 보여준다.

**애자일 자체는 방법론으로 보기 어렵다.** 그보다는 애자일이라는 특성을 지닌 다른 여러 방법론의 부모 격이라고 볼 수 있다.

**애자일은 소프트웨어 개발이 점증적으로 이루어져야 한다는 아이디어에 불을 붙였다. 애자일은 개발이 이루어지는 동안 요구사항 또한 바뀔 수 있고 오히려 바뀌어야 마땅하다고 본다.**

폭포수 개발에서는 요구사항 변화가 문제가 된다. 현실을 반영하지 못하는 이 제약을 해소하기 위해 나온 것이 바로 애자일이다.

### 스크럼

스크럼은 개발의 반복 주기마다 여는 **스프린트(sprint)**라고 부르는 회의를 까다로운 규범에 따라 정의한 정형화된 방법론이다.

스크럼에는 세 가지 주요 직책이 있다.

- 제품 책임자(product owner) - 고객의 소리를 전달하고 작업의 우선순위를 결정하는 역할을 한다.
- 개발팀
- 스크럼 마스터(Scrum master) - 스크럼 프로세스가 문제없이 진행될 수 있게 돕는, 팀의 코치 역할을 한다.

**소프트웨어 개발을 스프린트라고 부르는 작은 반복 주기로 나누는 것이 스크럼의 기본 아이디어다. 스프린트로 정해둔 기간 내에 해야 할 일의 양을 정해둔다. 그리고 각 스프린트를 마칠 때마다 나오는 결과를 점진적으로 고객에게 전달한다.**

개발해야 할 모든 기능을 제품 백로그(product backlog)에 넣어둔다. 제품 백로그에는 우선순위를 정해둔다. 각 스프린트마다 스프린트 백로그를 만들어 제품 백로그 항목 중에서 작업을 항목을 모아둔다. 스프린트는 보통 1~2주 정도로 나뉜다. 각 스프린트가 시작될 무렵 계획 회의를 연다.

매일 모든 팀원이 모여 자신이 진행하는 업무에 대해 아주 짧게 공유하는 스크럼 회의를 연다. 이 회의는 서서 진행한다. 각 팀원은 세 가지 질문에 답해야 한다.

1. 어제는 스프린트 달성에 도움이 될 만한 어떤 일을 했는가?
2. 오늘은 스프린트 달성에 도움이 될 만한 어떤 일을 할 것인가?
3. 스프린트 달성을 막는 장애물이 있는가?

마지막으로 지난 스프린트를 돌아보고 다음 스프린트에 대한 아이디어를 떠올리는 회고 회의를 연다.

제대로 수행하기만 한다면 스크럼은 소프트웨어를 매우 효과적으로 개발할 수 있게 해주는 좋은 방법이다. 하지만 현실에서는 제대로 구현하지 않는 경우가 많다. **스크럼이 실패하는 가장 큰 이유는 팀이나 개인의 헌신이 부족해서이다.**

### 칸반

칸반은 스크럼에 비해 훨씬 더 느슨하게 정의된 방법론이다. 칸반을 소프트웨어 개발에 적용할 때는 주로 **칸반 보드(Kanban board)**에 초점을 맞춘다.

칸반 모드는 몇 개의 칼럼이 있는 간단한 보드인데, 개발 프로세스가 진행되는 업무 단계를 표현한다. 프로젝트에서 해야 하는 일을 시각화하고, 동시에 진행하는 업무의 양을 제한하는 것이 핵심이다. 동시에 진행하는 업무를 WIP(Work in progress)라고 부른다.

칸반은 피드백 루프를 통해 끊임없이 더 나아지는 데 집중한다.

### 익스트림 프로그래밍

익스트림 프로그래밍 혹은 XP라고 불리는 이 방법론은 1996년 켄트 벡(Kent Beck)이 만들었다. XP는 단위 테스트, 테스트 주도 개발, 객체지향 프로그래밍, 고객 중심 등 당시의 모범 사례를 많이 가져와서 극단의 경지까지 끌어올렸다. 그래서 이름도 익스트림 프로그래밍이다.

**XP 프로젝트의 개발 프로세스는 아주 빡빡한 원칙을 중심으로 진행된다.**

XP는 TDD와 페어 프로그래밍에 의존한다.

### 다른 방법론과 비방법론

**어떤 방법론을 선택하느냐보다 반복과 측정이 가능한 프로세스를 갖췄느냐가 중요하다.**

## 28. 테스트와 QA 기초

자신의 작성하는 소프트웨어가 의도한 대로 바르게 작동하는 것도 개발자의 책임이다. 테스트의 기초를 모르면 '의도한 대로 바르게 작동한다'는 말이 어떤 뜻인지 제대로 이해하기 어렵다.

**테스트의 핵심 목표는 고객에게 부정적인 영향을 미칠 위험 요소를 줄이는 것이다.** 매우 단순한 애플리케이션이 아닌 이상에야 버그와 결함을 전부 찾아내거나 모든 입력을 테스트해보는 건 불가능하다. 소프트웨어의 어떤 영억이 가장 큰 위험 요소인지 우선순위를 정할 필요가 있다.

### 테스트 유형

- 블랙박스 테스트(black-box test) - 소프트웨어 내부가 보이지 않는 검은 상자라고 생각하는 진행하는 테스트다. 입력과 출력만 신경 쓴다.
- 화이트박스 테스트(white-box test) - 블랙박스 테스트와 반대이다. 시스템의 내부 구조를 이해하고 코드를 알고 있어야 제대로 해볼 수 있다.
- 인수 테스트(acceptance test) - 소프트웨어가 고객의 요구사항이나 기대에 맞게 제작되었는지 확인하는 테스트, 시스템을 전체적을 점검하는 테스트이다.
- 자동 테스트 - 테스트 실행이나 결과 검증이 자동으로 이루어지는 모든 테스트이다.
- 회귀 테스트(regression test) - 시스템이 과거에 작동했던 방식 그대로 작동하고 있는지 확인하는 테스트이다. 애자일에서는 새 기능이 기존 기능을 망가뜨릴 위험이 항상 도사리고 있기 때문에 회귀 테스트가 더욱 중요하다. 자동 테스트는 대부분 회귀 테스트다.
- 기능 테스트(functional test) - 시스템의 기능을 대상으로 진행하는 테스트이다. 기능 외에도 성능, 사용성, 회복성, 보안, 확장성 등 온갖 것을 대상으로 테스트할 수 있다. 기능 테스트는 기능적 측면에 초점을 맞춘다.
- 탐색적 테스트(exploratory test) - 테스트 케이스 없이 애플리케이션을 탐색하면서 예상치 못한 문제를 찾아본다. '게으름뱅이 테스트'라고 놀림받기도 하지만, 합리적인 테스트 케이스로는 알아내 어려웠을 버그를 찾아낼 수도 있다.

그 외의 테스트 유형은 다음과 같다.

- 부하 테스트(load test) - 과부하 상태에서 애플리케이션이 어떤 성능을 보이는지 확인
- 성능 테스트(performance test) - 특정 시나리오에서 애플리케이션의 성능을 확인
- 회복 테스트(recovery test) - 오류가 나거나 하드웨어에 문제가 생겼을 때 어떻게 회복하는지 확인
- 보안 테스트(security test)
- 스트레스 테스트(stress test)
- 사용성 테스트(usability test)

### 테스트 절차

1. 테스트 계획
2. 테스트 설계: 테스트 케이스 목록, 테스트 실행 조건의 유형
3. 테스트를 만들고 실행
4. 테스트 실행 결과를 평가
5. 버그나 결함은 수정하고 다시 테스트

### 애자일 팀의 테스트 방식

애자일 방식으로 개발할 때처럼 테스트 절차도 작은 단계로 나눌 필요가 있다. 기능 수준에서 테스트를 실행해야 한다. 각 기능을 미니 프로젝트라고 생각할 수 있다.

애자일에서는 자동 테스트가 더욱 중요하다.

### 테스트와 개발자

개발자라면 자신이 만든 코드의 품질에 그 누구보다 신경을 써야 한다. QA가 버그를 찾아낼 거라고 생각하지 마라. **버그는 늦게 발견될수록 고치는 데 더 많은 비용이 들기 때문이다.**

## 29. 테스트 주도 개발과 단위 테스트

### 단위 테스트

**코드를 가능한 한 작은 단위로 실행해보는 테스트를 작성하는 것을 단위 테스트라고 한다.**

단위 테스트는 회귀 테스트 역할도 한다.

단위 테스트는 절대적 요구사항이라고 생각하면 된다. 대부분의 객체지향 언어에서는 클래스 단위를 테스트한다.

단위 테스트와 자주 헷갈리는 개념으로 통합 테스트(integration test)가 있다. 통합 테스트는 클래스 이상의 것 혹은 더 큰 단위의 코드를 테스트한다.

### 단위 테스트의 가치

- **코드 설계가 개선된다.** 단위 테스트를 제대로 작성하려면 코드를 가능한 한 작은 단위로 고립시켜야 하는데, 그 과정에서 코드 설계의 문제점을 알게 된다. 이런 의미에서 **단위 테스트는 테스트보다는 평가 활동에 가깝다.**
- 자동화된 회귀 테스트를 만든다. 단위 테스트는 일종의 명세가 되기도 한다.

### 테스트 주도 개발(TDD)이란 무엇인가?

TDD의 기본 개념은 코드를 작성하기 전에 테스트부터 작성해서 그 코드가 해야 할 일을 명확하게 정의하는 명세로 쓰는 것이다.

**단위 테스트를 먼저 작성하고 그 테스트를 통과할 코드를 나중에 작성한다.**

단위 테스트가 처음에는 반드시 실패하며(빨간색), 테스트를 통과할 코드를 작성한 후(녹색)에 그 코드를 리팩터링하기 때문에, '빨간색, 녹색, 리팩터링'이라고 부르기도 한다.

### TDD의 목적은 무엇인가?

**TDD의 가장 큰 가치는 테스트를 명세로 쓸 수 있게 해주는 것이다.** 테스트는 거짓말을 하지 않기 때문에 훌륭한 명세이다.

**무엇을 테스트해야 할지 감이 오지 않는다면, 아직 대상을 제대로 이해하지 못한 것이다.**

TDD는 YAGNI 원칙을 지킬 수 있게 해준다.

## 30. 소스 제어

**소스 제어는 버전 제어라고도 부르며 파일의 버전과 소프트웨어 프로젝트의 소스 코드를 추적하는 방법을 가리킨다. 여러 개발자가 똑같은 파일을 가지고 작업할 때 이를 조율하는 역할을 한다.**

소스 제어가 중요한 이유는 다음과 같다.

- 코드를 분실할 가능성이 크게 낮아진다.
- 한 코드 베이스에 있는 여러 파일에서 함께 작업할 수 있다.

### 소스 제어의 기본

- 저장소: 코드를 저장해두는 장소
- 코드 체크아웃: 체크아웃한 코드가 로컬 버전이 되고 자신이 만든 수정 사항은 로컬 저장소에만 저장된다.
- 리비전: 소스 제어에 저장된 파일의 이진 버전. 수정 이전으로 돌아갈 때 사용한다.
- 분기: 코드를 나무라고 생각하면, 나무의 몸통인 트렁크에서 가지인 브랜치가 여러 개 나온다.
- 병합: 어떤 코드에 있는 변경 사항을 다른 코드에 합치는 것
- 충돌: 같은 파일의 같은 부분에서 동시에 수정이 생겼을 때 발생한다. 컴퓨터는 어떤 수정 사항이 우선인지 모른다.

### 기술

크게 중앙 집중형과 분산 제어형으로 분류된다. 중앙 집중형이 더 오래되었다.

중앙 집중형 소스 제어 시스템(CVCS)의 중앙 서버에는 한 개의 저장소가 있다. 업무 흐름은 다음과 같다.

1. 작업 중인 코드의 로컬 사본을 저장소로부터 업데이트하기
2. 수정하기
3. 중앙 저장소에 수정 사항 커밋하기 (그리고 충돌 해결하기)

분산 제어형 소스 제어 시스템(DVCS)은 각 개발자가 자신의 컴퓨터에 전체 저장소의 완전한 사본을 가지고 있는다. 하지만 현실적으로는 저장소 중앙 버전 혹은 마스터 저장소는 있어야 한다. 풀을 사용하여 로컬 저장소에 수정 사항을 내려받고, 자신이 수정한 사항을 마스터 저장소에 푸시한다.

CVS와 서브버전 중앙 집중형 소스 제어 시스템에 속하고, 깃과 머큐리얼이 분산 제어형 소스 제어 시스템에 속한다.

### 가장 인기 있는 소스 제어 시스템에 대한 간단한 소개

- CVS: 서브버전의 전신
- 서브버전: 저자가 가장 많이 사용했다.
- 깃: 소스 제어의 동의어가 되었다.
- 머큐리얼: 깃이 맥가이버라면 머큐리얼은 제임스 본드다.

## 31. 지속적 통합

지속적 통합(CI, continuous integration)은 소프트웨어를 빌드하고 , 테스트하고, 패키징하는 과정을 자동화해준다.

지속적 통합은 개발자들이 각기 작업 중인 코드를 더 자주 병합시키기 때문에 병합 지옥에 빠지지 않게 막아준다. 그리고 빠른 피드백을 제공하기 때문에 애자일 개발에서 매우 중요한 요소이다.

### 과거의 코드 빌드 방법

개발자들은 자신의 컴퓨터에서 소프트웨어를 빌드하는 자신만의 방법을 만들었다. 이런 방식은 똑같은 코드를 가지고 완전히 다른 소프트웨어를 만들어낼 수 있다는 큰 문제가 존재한다.

다른 버전의 외부 라이브러리가 설치되어 있거나, 최신 버전의 코드가 아니거나, 파일이나 폴더 구조가 다를 수도 있다.

초창기에는 이러한 문제를 해걀하기 위해 빌드 서버를 도입하곤 했다. 각자의 컴퓨터에서 소프트웨어를 빌드하는 대신 환경과 라이브러리를 갖춘 중앙 빌드 서버를 만드는 것이다.

처음에는 주간 빌드로 시작했다. 그러다 야간 빌드가 인기를 끌었다.

### 마침내 지속적 통합으로

지속적 통합은 체크인하면 단위 테스트, 정적 코드 분석기 같은 코드 품질 평가 실행이 동시에 이루어지는 수준까지 진화했다. 개발자들이 코드를 최대한 빨리 자주 체크인하게 하는 부분이 가장 어려웠다.

### 지속적 통합 작업 흐름

1. 코드 체크인
2. 새 빌드 시작: CI 서버가 모니터링하던 소스 제어 브랜치의 변화를 감지하고 새로운 빌드를 시작한다.
3. 코드 체크아웃
4. 코드 컴파일빌드 스크립트가 소스 코드 빌드 명령을 실행한다.
5. 정적 분석기 실행: 코드 품질(상속의 깊이, 코드 라인 수 등)을 측정한다.
6. 단위 테스트 실행
7. 빌드 결과 보고
8. 소프트웨어 패키징
9. 코드의 선택적 배포(지속적 배포)
10. 완료

### CI 서버와 소프트웨어

- 젠킨스(Jenkins): 원래는 자바 환경에서 CI를 할 수 있게 만들어진 자바 프로그램이었다. 자체적으로 웹서버를 갖추고 있어서 설치하고 실행하기가 쉽고 플러그인도 많다.
- 허드슨(Hudson): 허드슨에서 젠킨스가 갈라져 나왔다. 허드슨은 오라클이 관리한다.
- 트래비스 CI(Travis CI): 호스팅 또는 서비스 형태로 제공된다.
- TFS(Team Foundation Server): 마이크로소프트 솔루션이 필요한 사람에게는 잘 맞을 수 있다.
- 팀시티(TeamCity): 젯브레인스에서 만들었다.

## 32. 디버깅

**개발자가 되면 코드 디버깅에 아주 많은 시간을 들이게 될 것이다.**

**디버깅에 접근하는 태도가 무엇보다 중요하다.**

### 디버깅이란 무엇인가?

코드 베이스에서 문제의 근원을 찾아서 그 문제를 일으킬 만한 요인을 가려내고, 여러 가설을 시험해보는 과정을 통해 뿌리가 되는 진정한 원인을 찾고 그 원인을 제거한 후, 다시는 그 문제가 일어나지 않도록 하는 것이 디버깅이다.

### 디버거를 쓰지 마라

디버거는 최후의 수단으로 남겨둬야 한다.

### 오류를 재현하라

재현할 수 없는 문제는 디버깅하는 게 의미가 없다. 버그를 재현해서 진짜 버그가 발생한 게 맞는지 확인부터 해야 한다.

### 앉아서 생각하라

문제를 일으킬 만한 원인이 무엇일지 생각하라. 시스템이 어떻게 작동하는지, 무엇이 이상을 일으켰을지 생각해보라.

**적어도 2~3가지 가설을 세워야 한다.**

### 가설을 테스트하라

자신이 세운 가설을 테스트할 단위 테스트를 작성하라. 단위 테스트를 작성하고 하나씩 통과할 때마다 가능성을 제거하는 것이다.

잘못된 경로를 계속해서 다시 방문하게 하는 게 디버거의 아주 큰 단점이다. 단위 테스트를 쓰는 게 불가능한 경우에만 디버거를 사용한다. 다만 디버거를 쓰는 목적을 분명히 해야 한다.

### 가정을 확인하라

사람은 코드가 특정한 방식으로 작동한다고 가정하는 경향이 있다. 그럴 때는 가정을 확인하라.

### 분할 정복하라

때로는 문제를 반으로 자를 방법을 찾아보는 것도 좋다.

### 고칠 때는 이유를 이해하라

문제가 왜 고쳐졌는지 이해하지 못하면 아직 문제를 고친 것이 아니다.

### 예술과 과학

개발이 그렇듯 디버깅도 예술과 과학의 영역에 속하는 활동이다. **디버깅을 잘하려면 연습해야 한다.**

## 33. 코드 유지 보수

**새 코드를 쓰는 시간보다 코드를 유지 보수하는 시간이 훨씬 더 길다.**

따라서 다음 두 가지를 익혀두면 좋다.

- 코드를 적절히 유지 보수하는 방법
- 유지 보수하기 쉽게 코드를 작성하는 방법

개발자는 어떤 형태로든 코드 유지 보수 업무를 하게 될 것이다.

### 보이 스카우트 규칙

보이 스카우트 규칙이 코드 유지 보수에 도움이 된다.

**캠핑한 장소를 자신이 처음 왔을 때보다 더 깨끗하게 만들어 두고 떠나라.**

### 가독성이 가장 중요하다

코드를 유지 보수하기 쉽게 만드는 데 가장 큰 영향을 미치는 요소는 가독성이다.

### 코드 리팩터링

리팩터링은 기존 코드 설계를 개선하는 과정이다. 기존 코드의 기능은 그대로 두고 조금 더 가독성을 높이는 것이다.

### 자동화는 필수다

자동화는 프로젝트의 원활한 유지 보수에 필수 요소다.

### 주석을 쓸 거라면 잘 써라

주석이 없어도 이해할 수 있는 명확하고 표현력이 좋은 코드를 쓰는 게 좋다. 하지만 주석을 쓸 거라면 꼭 제대로 써야 한다. 수수께끼 같은 주석은 수수께끼 같은 코드만큼 나쁘다.

커밋 메시지도 최대한 알아보기 쉽고 도움이 되게 써라.

## 34. 직업과 직함

직함은 그리 중요하지 않다. 하지만 이직 시에는 좋은 직함을 가진 사람이 더 좋은 직장을 구할 확률이 높다.

### 피해야 할 직함

막 프로그래머가 된 많은 이들은 자신이 주니어 개발자라는 직함을 달아야 한다고 생각한다. 하지만 사실 그렇지 않다.

### 기본적인 직함 혹은 직급

- 소프트웨어 개발 테스트 엔지니어(Software Development Engineer in Test, SDET): 테스트 코드를 작성하거나 개발팀 직원이 더 쉽게 일할 수 있게 도와주는 도구를 만든다.
- 개발자(주니어/표준/시니어)
- 팀 리드 또는 기술 리드: 팀을 이끄는 동시에 개발 업무와 관리 업무도 한다.
- 아키텍트(Architect): 전체 시스템을 설계하고 기술과 아키텍처, 실험적인 프로토타입 기능이나 전체 시스템을 결정하는 회의에 참석한다.
- 디렉터(director): 연구원(fellow)이나 기술 연구원(technical fellow)이라고 부르기도 한다.

## 35. 업무 유형

소프트웨어 개발자들은 코드 작성 업무 외에도 일이 많다.

- 코드 작성하기
- 버그 고치기
- 설계와 아키텍처: 훌륭한 소프트웨어 개발자치고 바로 코딩부터 하는 사람은 많지 않다.
- 회의
- 학습: 학습은 직장에서도 해야 하고 자유 시간에도 해야 한다. 저자는 매일 아침 30분씩 소프트웨어 개발 블로그를 찾아보고 최신 지식을 살펴봤다.
- 실험과 탐색: 기존의 코드를 읽는 데 많은 시간을 들여야 한다. 그리고 이것저것 해보아야 할 때가 있다. 새로운 API나 기술을 써보기 위해 샘플 프로그램을 작성해야 할 때가 있다.
- 테스트
- 생각하기: 코드를 작성하기 전에 코드를 어떻게 작성할지 적어도 **세 번** 이상 생각해보는 것이 좋다.
- 고객/이해 당사자와 소통하기: 자신이 만드는 시스템의 요구 사항을 잘 이해해야만 좋은 개발자가 될 수 있다.
- 교육/멘토링: 경험 많은 훌륭한 개발자는 개발팀 전체의 능력을 끌어올릴 수 있다.
