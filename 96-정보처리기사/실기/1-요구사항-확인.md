# 1장: 요구사항 확인

## 1섹션: 소프트웨어 생명 주기

**소프트웨어 생명 주기(Software Life Cycle)**는 소프트웨어 개발 과정을 각 단계별로 나눈 것이다.

대표적인 생명 주기 모형은 다음과 같다.

- **폭포수 모형(Waterfall Model)**: 각 단계를 확시맇 매듭짓고 다음 단계를 진행하는 개발 방법론이다.
- **프로토타입 모형(Prototype Model)**: 견본품을 만들어 최종 결과물을 예측하는 모형이다.
- **나선형 모형(Spiral Model)**: 여러 번의 소프트웨어 개발 과정을 거쳐 점진적으로 개발하는 모형이다.
  - **계획 → 위험 분석 → 개발 → 평가**
- **애자일 모형(Agile Model)**: 요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하면서 개발하는 모형이다.
  - 스크럼
  - XP
  - 칸반
  - Lean
  - 기능 중심 개발(FDD)

**소프트웨어 공학(SE)**은 소프트웨어의 위기를 극복하기 위한 방안으로 연구된 학문이다.

## 2섹션: 스크럼 기법

**스크럼(Scrum)**은 팀이 중심이 되어 개발의 효율성을 높이는 기법이다.

스크럼 팀은 다음과 같이 구성된다.

- **제품 책임자(PO; Product Owner)**: 백로그를 작성하는 주체
- **스크럼 마스터(SM)**
- **개발팀(DT)**: PO와 SM을 제외한 모든 팀원

스크럼 개발 프로세스는 다음과 같다. 순서를 기억해야 한다.

1. 스프린트 계획 회의
2. 스프린트: 2~4주 단위
3. 일일 스크럼 회의
4. 스프린트 검토 회의
5. 스프린트 회고

## 3섹션: XP 기법

XP(eXtreme Programming)는 요구사항에 유연하게 대응하기 위해 고객의 참여와 개발 과정의 반복을 극대화하여 개발 생산성을 향상시키는 방법이다. 릴리즈의 기간을 짧게 반복하여 요구사항 반영에 대한 가시성을 높인다.

XP의 5가지 핵심 가치는 다음과 같다.

- **단순성**
- **의사소통**
- **존중**
- **피드백**
- **용기**

XP 개발 프로세스는 다음과 같다. 순서를 기억해야 한다.

1. 릴리즈 계획 수립
2. 이터레이션(주기)
3. 승인 검사(인수 테스트)
4. 소규모 릴리즈

XP의 주요 실천 방법은 다음과 같다.

- Pair Programming
- Collective Ownership
- TDD
- Whole Team
- Continuous Integration
- **Rafactoring(리팩토링)**: 기능의 변경 없이 시스템을 재구성하는 것이다. 리팩토링의 목적은 프로그램을 쉽게 이해하고 쉽게 수정하여 빠르게 개발하는 것이다.
- Small Releases

## 5섹션: 요구사항 정의

**요구사항**은 어떤 문제를 해결하기 위해 필요한 제약조건을 의미한다.

요구사항의 유형은 다음과 같다.

- **기능 요구사항**
- **비기능 요구사항**: 품질이나 제약사항과 관련된 요구사항
- 사용자 요구사항
- 시스템 요구사항: 개발자 관점

## 6섹션: 요구사항 개발 프로세스

요구사항 개발 프로세스는 다음과 같다.

1. **도출**
2. **분석**
3. **명세**
4. **확인**

요구사항을 도출하는 주요 기법은 다음과 같다.

- 청취와 인터뷰
- 설문
- 브레인스토밍
- 워크샵
- 프로토타이핑
- 유스케이스

요구사항 분석에 사용되는 도구는 다음과 같다.

- 자료 흐름도(DFD)
- 자료 사전(DD)

## 7섹션: 요구사항 분석

요구사항을 이해하고 문서화하는 활동을 의미한다.

**구조적 분석 기법**은 자료의 흐름과 처리를 중심으로 하는 요구사항 분석 방법이다. 도구는 다음과 같다.

- 자료 흐름도(DFD)
- 자료 사전(DD)
- 소단위 명세서
- 개체 관계도(ERD)
- 상태 전이도(STD)
- 제어 명세서

### 자료 흐름도

**자료 흐름도(DFD; Data Flow Diagram)**는 자료의 흐름 및 변환 과정과 기능을 도형 중심으로 기술하는 방법이다.

자료 흐름도의 기본 기호는 다음과 같다.

- **프로세스(Process)**: 원
- **자료 흐름(Data Flow)**: 화살표
- **자료 저장소(Data Store)**: 좌우가 열린 사각형
- **단말(Terminator)**: 사각형

### 자료 사전

**자료 사전(DD; Data Dictionary)**은 자료 흐름도에 있는 자료를 더 자세히 정의하고 기록한 것이다. 메타 데이터라고도 한다.

자료 사전의 표기 기호는 다음과 같다.

| 기호  | 의미 |
| ----- | ---- |
| `=`   | 정의 |
| `+`   | 연결 |
| `( )` | 생략 |
| `[ ]` | 선택 |
| `{ }` | 반복 |
| `* *` | 설명 |

## 9섹션: UML의 개요

**UML(Unified Modeling Language)**은 소프트웨어 개발 참여자 간의 의사소통이 원활하게 이루어지도록 표준화된 대표적인 객체지향 모델링 언어이다.

UML의 구성 요소는 다음과 같다.

- **사물**
- **관계**
- **다이어그램**

### 사물

**사물(Things)**은 다이어그램 안에서 관계가 형성될 수 있는 대상들을 말한다.

사물의 종류는 다음과 같다.

- **구조 사물**: 시스템의 개념적, 물리적 요소
- **행동 사물**: 시간과 공간에 따른 요소들의 행위
- **그룹 사물**: 요소들을 그룹으로 묶음
- **주해 사물**: 부가적인 설명이나 제약조건 등

## 10섹션: UML - 관계

**관계(Relationship)**는 사물과 사물 사이의 연관성을 표현하는 것이다.

### 연관 관계

- 연관(Association) 관계는 2개 이상의 사물이 서로 관련되어 있는 관계이다.
- 방향성은 화살표로 표현한다.
- 다중도를 선 위에 표기한다.

### 집합 관계

- 집합(Aggregation) 관계는 하나의 사물이 다른 사물에 포함되어 있는 관계이다.
- 서로 독립적이다.
- 속이 빈 마름모로 표현한다.
- 예) 컴퓨터와 프린터

### 포함 관계

- 포함(Composition) 관계는 포함하는 사물의 변화가 포함되는 사물에게 영향을 미치는 관계이다.
- 서로 독립될 수 없다.
- 속이 채워진 마름모로 표현한다.
- 예) 문과 열쇠

### 일반화 관계

- 일반화(Generalization) 관계는 하나의 사물이 다른 사물보다 더 일반적이거나 구체적인 관계이다.
- 속이 빈 화살표로 표현한다.
- 예) 커피와 아메리카노

### 의존 관계

- 의존(Dependency) 관계는 서로에게 영향을 주는 짧은 시간 동안만 연관을 유지하는 관계이다.
- 점선 화살표로 표현한다.
- 예) 할인율과 등급

### 실체화 단계

- 실체화(Realiazation) 관계는 할 수 있는 기능으로, 서로를 그룹화할 수 있는 관계이다.
- 빈 점선 화살표로 표현한다.
- 예) 비행기와 '날 수 있다'

## 11섹션: UML - 다이어그램

**다이어그램(Diagram)**은 사물과 관계를 도형으로 표현한 것이다.

**정적 모델링에서는 구조적 다이어그램을 사용하고, 동적 모델링에서는 행위 다이어그램을 사용한다.**

**구조적 다이어그램**의 종류는 다음과 같다.

- **클래스 다이어그램**
- 객체 다이어그램: 럼바우 객체지향 분석 기법에 사용됨
- **컴포넌트 다이어그램**
- 배치 다이어그램: 요소들의 위치를 표현함
- 복합체 구조 다이어그램
- 패키지 다이어그램

구현 단계에서 사용하는 것은 컴포넌트 다이어그램과 배치 다이어그램이다.

**행위 다이어그램**의 종류는 다음과 같다.

- **유스케이스 다이어그램**: 사용자 관점에서 표현함
- **순차 다이어그램**: 시스템이나 객체들이 주고받는 메시지를 표현함
- 커뮤니케이션 다이어그램
- **상태 다이어그램**: 럼바우 객체지향 분석 기법에 사용됨
- **활동 다이어그램**: 객체의 처리 로직이나 조건에 따른 처리의 흐름을 순서에 따라 표현함
- 상호작용 개요 다이어그램
- 타이밍 다이어그램

**스테레오 타입(Stereotype)**은 UML에서 표현하는 기본 기능 외에 추가적인 기능을 표현하는 것이다. 겹화살괄호(<< >>)를 이용한다.

## 14섹션: 클래스 다이어그램

**클래스 다이어그램**은 클래스, 클래스가 가지는 속성, 클래스 사이의 관계를 표현한 것이다.

책에서 예를 참고한다.

**연관 클래스**는 연관 관계에 있는 두 클래스에 추가적으로 표현해야 할 속성이나 오퍼레이션이 있는 경우 생성하는 클래스이다.

## 18섹션: 패키지 다이어그램

**패키지 다이어그램**은 요소들을 그룹화한 패키지 간의 의존 관계를 표현한 것이다.

## 20섹션: S/W 공학의 발전적 추세

- **소프트웨어 재사용**: 이미 개발된 소프트웨어를 다른 소프트웨어 개발이나 유지에 사용하는 것이다.
  - 합성 중심
  - 생성 중심: 명세를 구체화
- **소프트웨어 재공학**: 기존 시스템을 이용하여 보다 나은 시스템을 구축하는 것이다. 유지보수의 생산성을 향상시킨다.
- **CASE(Computer Aided Software Engineering)**: 요구 분석, 설계, 구현, 검사 과정 전체 또는 일부를 전용 소프트웨어 도구를 사용하여 자동화하는 것이다.

## 22섹션: 비용 산정 기법 - 상향식

LOC(원시 코드 라인 수, source Line Of Code) 기법은 각 기능의 원시 코드 라인 수의 비관치, 낙관치, 기대치를 측정하여 예측치를 구하고 이를 이용하여 비용을 산정하는 기법이다.
$$
예측치=\frac{낙관치+4·기대치+비관치}{6}
$$
산정 공식은 다음과 같다.

- **노력(인월) = 투입 인원 × 개발 기간**

## 23섹션: 수학적 산정 기법

수학적 산정 기법은 상향식 비용 산정 기법이다.

### COCOMO 모형

LOC에 의한 비용 비용 산정 기법이다.

소프트웨어 개발 유형은 다음과 같다.

- **조직형(Organic Mode)**: 5만 라인 이하인 중·소 규모
- **반분리형(Semi-Detached Mode)**: 30만 라인 이하인 중간 규모
- **내장형(Embedded Mode)**: 30만 라인 이상인 초대형 모드

### Putnam 모형

**소프트웨어 생명 주기**의 전 과정 동안에 사용될 노력의 분포를 가정해 주는 모형이다. Rayleigh-Norden 곡선의 노력 분포도를 이용한다.

### FP 모형

기능 점수(FP; Function Point) 모형은 기능 점수를 구한 후 이를 이용하여 비용을 산정하는 기법이다.

## 24섹션: 프로젝트 일정 계획

- **PERT**(Program Evaluation and Review Technique, 프로그램 평가 및 검토 기술): 프로젝트에 필요한 전체 작업의 상호 관계를 표시하는 네트워크로 낙관치, 기대치, 비관치로 나누어 각 단게별 종료 시기를 결정하는 방법이다.
- **CPM**(Critical Path Method, 임계 경로 기법): 프로젝트 완성에 필요한 작업을 나열하고 작업에 필요한 소요 기간을 예측하는데 사용하는 기법이다. 임계 경로는 최장 경로를 의미한다.
- 간트 차트: 프로젝트의 각 작업들이 언제 시작하고 언제 종료되는지에 대한 작업 일정을 도표를 이용하여 표시하는 프로젝트 일정표이다.

## 26섹션: 소프트웨어 개발 표준

### CMMI

**CMMI**(Capability Maturity Model Integration)는 개발 조직의 업무 능력 및 성숙도를 평가하는 모델이다.

소프트웨어 프로세스 성숙도는 다음과 같이 5단계로 구분한다.

1. **초기(Initial)**
2. **관리(Managed)**
3. **정의(Defined)**
4. **정량적 관리(Quantitatively Managed)**
5. **최적화(Optimizing)**

### SPICE

**SPICE**(Software Process Improvement and Capability dEtermination)는 소프트웨어 프로세스를 평가 및 개선하는 국제 표준이다.

## 28섹션: 소프트웨어 개발 프레임워크

**소프트웨어 개발 프레임워크**는 소프트웨어 개발에 공통적으로 사용되는 구성 요소와 아키텍처를 일반화하여 여러 가지 기능들을 제공해주는 **반제품 형태의 소프트웨어 시스템**이다.

종류는 다음과 같다.

- 스프링 프레임워크
- 전자정보 프레임워크
- 닷넷 프레임워크

특성은 다음과 같다.

- **모듈화**(Modularity)
- **재사용성**(Reusability)
- **확장성**(Extensibility)
- 제어의 역흐름(Inversion of Control)
